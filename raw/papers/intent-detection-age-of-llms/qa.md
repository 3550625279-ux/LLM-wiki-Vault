### Q1: 什么是任务导向对话系统（TODS）中的意图检测？

<details>
<summary>查看答案</summary>

意图检测是 TODS 自然语言理解层的核心组件，负责将用户在每个对话轮次的查询映射到系统可执行的动作。例如，用户说"帮我查一下订单状态"，系统需要识别出这是一个"查询订单"的意图，然后触发相应的 API 调用或流程。

在意图检测中，系统需要：
1. 将查询映射到已知意图（in-scope 分类）
2. 拒绝不属于任何已知意图的查询（OOS 检测）

意图的范围从"非常宽泛"（如"产品推荐"）到"非常具体"（如"根据电池寿命推荐笔记本电脑"）。

</details>

### Q3: SetFit 是什么？它的两阶段训练过程是怎样的？

<details>
<summary>查看答案</summary>

SetFit（Sentence Transformer Fine-Tuning）是一种高效的少样本学习方法，不需要提示模板即可微调句子转换器。

**两阶段训练**：

**阶段 1 — 对比微调**：
- 使用 Siamese 网络结构
- 对句子对进行对比学习
- 目标：同一意图的句子表示靠近，不同意图的远离
- 基础模型：MPNet（all-mpnet-base-v2）

**阶段 2 — 分类头训练**：
- 使用阶段 1 微调后的编码器生成句子表示
- 在表示上训练一个线性分类头
- 使用 Sigmoid 激活函数（支持多标签分类）

SetFit 的优势是训练数据需求少（少样本）、推理速度快（p50 延迟仅 30ms），但准确率不如 LLM。

</details>

### Q4: 负面数据增强（Negative Data Augmentation）是什么？如何实现？

<details>
<summary>查看答案</summary>

负面数据增强是一种通过生成"看似合理但实际上是 OOS"的训练样本来提升 SetFit OOS 检测能力的方法。

**实现步骤**：

1. **关键词提取**：使用 KeyBERT 从每个训练样本中提取关键词
2. **样本变换**：对每个关键词，随机选择：
   - 50% 概率：完全删除该关键词
   - 50% 概率：替换为 5 个随机字符的字符串
3. **标记为 OOS**：将变换后的样本标记为 OOS
4. **控制数量**：增强样本数 = 0.2 × 原始训练集大小

**示例**：
- 原始："looking for a gaming laptop"
- 增强："looking for a"（删除了 gaming laptop）
- 增强："looking for a XYCVD QSDER"（替换为随机串）

**为什么有效**：这些增强样本与原始样本有相似的词汇模式，迫使模型不能仅靠关键词匹配来分类，从而学习更鲁棒的决策边界。实验显示 F1 提升 >5%。

</details>

## 🟡 中级题（5 题）

### Q6: 混合系统中的 Monte Carlo Dropout 路由机制是如何工作的？

<details>
<summary>查看答案</summary>

混合系统的核心是使用 MC Dropout 估计 SetFit 的预测不确定性，只在不确定时才调用 LLM。

**详细流程**：

1. **开启 Dropout**：在 SetFit 的 hidden layers 和 attention layers 设置 dropout=0.1
2. **多次采样**：对同一查询进行 M 次前向传播（每次 Dropout 掩码不同），得到 M 个预测
3. **计算不确定性**：
   - 统计 M 次预测中不同标签的数量（unique_labels）
   - 如果 unique_labels > 1 **或** unique_labels < M/2 → 不确定
   - 否则 → 确定
4. **路由决策**：
   - 确定 → 返回 SetFit 结果（~30ms）
   - 不确定 → 调用 LLM 重新预测（~1.7-3.9s）

**理论基础**：Gal & Ghahramani (2016) 证明 MC Dropout 可以近似贝叶斯神经网络的后验推断，因此预测方差可以作为不确定性估计。

**M 的选择**：实验中 M ∈ {5, 10, 20}，结果表明 M=10 是较好的平衡点（边际收益在 M>10 后变小）。

</details>

### Q7: 为什么 LLM 在 OOS 检测上表现差？控制实验揭示了什么规律？

<details>
<summary>查看答案</summary>

**核心发现**：LLM 的 OOS 检测能力受两个因素显著影响：

**因素 1 — 意图标签范围（Scope of Intents）**：
- 范围越大（S 增加），OOS 检测越差
- 当意图覆盖广泛的语义空间时，LLM 更倾向于把查询"往里塞"（匹配到某个宽泛意图），而不是判断为 OOS
- 这类似于"万金油效应"——标签越宽泛，越容易匹配

**因素 2 — 标签空间大小（Number of Labels）**：
- 标签越多（L 增加），OOS 检测退化越严重
- 标签空间增大时，LLM 更难对查询说"不"
- OOS AUCROC 的退化比 In-Scope Accuracy 更严重

**与 SetFit 的对比**：
- SetFit 在两个因素变化时都更稳定（虽然也退化，但幅度小得多）
- 这说明传统模型对 class design 的鲁棒性更好

**实际意义**：
- 设计意图体系时应倾向于更细粒度的标签
- 控制标签空间大小
- 这对 LLM 友好的 TODS 设计有重要指导意义

</details>

### Q8: 两步 OOS 检测方法的每一步分别做什么？为什么分开比一起做好？

<details>
<summary>查看答案</summary>

**Step 1 — 纯 In-Scope 预测**：
- 修改 Prompt，只包含 in-scope 标签（完全移除 OOS 选项）
- LLM 被迫从已知标签中选择一个
- 即使是 OOS 查询，也会被分配到某个 in-scope 标签

**Step 2 — 内部表示相似度比较**：
- 获取用户查询的 LLM 内部表示（decoder 层最后一个 prompt token 的隐藏状态）
- 与 Step 1 预测意图的所有训练样本表示比较
- 使用余弦相似度，取均值
- 如果相似度低于阈值 → 判定为 OOS

**为什么分开更好？**

1. **任务解耦**：LLM 在生成模式下不擅长同时做"选择"和"拒绝"。分开后，每个任务更简单
2. **减少标签空间**：Step 2 只需比较与预测意图的相似度，而非与所有意图比较
3. **内部表示信息更丰富**：编码了 LLM 对输入的"理解程度"，比生成文本更细腻
4. **避免 LLM 偏见**：LLM 在生成模式下倾向于输出已知标签而非拒绝（"倾向于说 Yes"）

**实验结果**（Mistral-7B）：
- OOS Recall 从 0.465 → 0.715（SOFMattress，+53%）
- 准确率和 F1 整体提升 >5%
- 额外延迟 ~300ms（只需一次前向传播）

</details>

### Q9: 本文的混合系统与纯 LLM 方案相比，在延迟和准确率上各有什么权衡？

<details>
<summary>查看答案</summary>

**延迟-准确率权衡数据**（不包括 OADP）：

| 方案 | 平均 F1 | 平均延迟 | 性能差 | 延迟分数 |
|------|---------|----------|--------|---------|
| Claude v3 Haiku (纯) | 0.760 | 2.345s | 基准 | 100% |
| SNA + Haiku (M=10) | 0.737 | 1.005s | -2.3% | 43% |
| Mistral Large (纯) | 0.762 | 3.867s | 基准 | 100% |
| SNA + Mistral-L (M=10) | 0.736 | 1.453s | -2.5% | 38% |

**关键权衡点**：

1. **性能损失可控**：混合系统仅比纯 LLM 低 ~2%（不含 OADP）/ ~4%（含 OADP）
2. **延迟大幅降低**：减少 ~50-60% 的延迟
3. **M 增大的边际收益递减**：M=5 → M=10 提升约 1.5%，M=10 → M=20 仅提升 0.1%
4. **OOD 泛化是短板**：对 OADP（分布外数据），混合系统的性能差距更大（~4%），因为 SetFit 的泛化能力差
5. **成本角度**：LLM 调用次数减少 50%+，直接降低推理成本

**实际选择**：
- 延迟敏感的在线服务 → 混合系统（M=10）
- 准确率优先且可接受延迟 → 纯 LLM
- 预算有限 → 纯 SetFit + 负面增强

</details>

### Q10: 论文中提到的"标签名随机化"技巧是为什么？有什么实际影响？

<details>
<summary>查看答案</summary>

**做法**：将意图标签名（如"Product Recommendation"、"Order Tracking"）随机替换为无意义的名称（如"Label-23"、"Label-7"）。

**为什么要做这个？**

LLM 具有丰富的世界知识，会利用标签名的表面语义来辅助预测。例如：
- 标签名"Product Recommendation"本身就暗示了查询应该与推荐相关
- LLM 可能基于标签名语义进行匹配，而非基于训练示例的真实模式

这会导致：
1. **评估偏差**：LLM 看起来很准，但实际上是在"作弊"——利用了标签名的语义信息而非学习意图本身
2. **不公平对比**：SetFit 看不到标签名，LLM 看得到，对比不公平
3. **对新意图泛化差**：真实场景中新意图的名称可能不那么直觉

**实际影响**：
- 在开源数据集上尤其重要（标签名可能是公开的语义名称）
- 在内部数据集上影响较小（标签名可能已经是内部代号）
- 这是一个实验设计的"最佳实践"，确保评估的公平性

</details>

### Q11: 从贝叶斯视角，MC Dropout 不确定性估计的理论基础是什么？它有什么局限？

<details>
<summary>查看答案</summary>

**理论基础**：

Gal & Ghahramani (2016) 证明了在标准 Dropout 训练的网络上进行多次前向传播（每次使用不同的 Dropout 掩码），可以近似变分贝叶斯推断：

- 每次 Dropout 前向传播 ≈ 从后验分布 q(θ) 中采样一次
- M 次采样的均值 ≈ 后验预测均值
- M 次采样的方差 ≈ 预测不确定性（认知不确定性，epistemic uncertainty）

**形式化表达**：

```
p(y|x, D) ≈ (1/M) Σ p(y|x, θ_m), θ_m ~ q(θ)

不确定性 = Var[p(y|x, θ_m)]
```

**在本文中的应用**：
- 不是用概率方差，而是用预测标签的**一致性**来估计不确定性
- 如果 M 次预测的标签高度一致 → 确定
- 如果标签分散 → 不确定

**局限性**：

1. **仅捕获认知不确定性**：MC Dropout 只估计模型参数的不确定性，不捕获数据固有的噪声（偶然不确定性，aleatoric uncertainty）
2. **需要足够大的 M**：M 太小（如 M=3）会导致不确定性估计不稳定
3. **Dropout 率需要调优**：p=0.1 是经验值，不同任务可能需要不同值
4. **假设近似有效**：MC Dropout 是一个粗糙的近似，真正的贝叶斯推断更复杂
5. **推理成本线性增加**：M 次前向传播 = M 倍计算量

</details>

### Q12: 如果你要将本文的混合系统部署到生产环境，需要考虑哪些额外因素？

<details>
<summary>查看答案</summary>

**工程层面**：

1. **缓存策略**：
   - 相同/相似查询可以缓存 LLM 结果
   - 论文提到缓存可以降低延迟但未深入讨论
   - 需要设计相似查询的匹配机制

2. **LLM 调用的容错**：
   - LLM API 可能超时/限流/出错
   - 需要 fallback 策略（超时时默认使用 SetFit 结果？拒绝回答？）

3. **延迟 SLA**：
   - 混合系统的延迟分布是双峰的（快路径 ~30ms + 慢路径 ~1.7s）
   - p99 延迟可能远高于 p50
   - 需要根据业务 SLA 设置路由阈值

4. **成本监控**：
   - LLM 调用成本随路由比例变化
   - 需要监控"路由率"（被路由到 LLM 的查询比例）

**模型层面**：

5. **意图体系更新**：
   - 新增意图时，SetFit 需要重新训练
   - LLM 只需更新 Prompt 中的意图描述和 ICL 示例
   - 混合系统需要同步更新两边

6. **数据漂移监控**：
   - 用户查询分布变化可能导致 SetFit 不确定性分布变化
   - 路由阈值可能需要定期重新校准

7. **公平性**：
   - 不同类型的查询路由到不同模型可能产生不一致的体验
   - 需要确保路由不会对特定用户群体产生系统性偏差

**运维层面**：

8. **MC Dropout 的 GPU 内存**：M 次前向传播需要考虑内存
9. **模型版本管理**：SetFit 和 LLM 的版本需要协同更新
10. **A/B 测试框架**：比较混合系统与纯 LLM 的线上效果

</details>

### Q13: 本文的内部表示 OOS 检测方法有什么潜在的改进方向？

<details>
<summary>查看答案</summary>

**当前方法的局限和改进方向**：

**1. 表示提取位置**：
- 当前：使用最后一个 prompt token 的表示
- 改进：探索不同层的表示（浅层 vs 深层可能编码不同信息）
- 改进：使用多个 token 的表示（如所有 prompt token 的平均池化）
- 改进：动态选择最佳层（类似 ELMo 的方法）

**2. 相似度度量**：
- 当前：余弦相似度 + 均值
- 改进：使用马氏距离（考虑表示空间的分布形状）
- 改进：使用最近邻距离（而非均值，对异常值更鲁棒）
- 改进：训练一个小的二分类器代替简单阈值

**3. 阈值设定**：
- 当前：全局统一阈值
- 改进：每个意图设置不同的阈值（不同意图的表示分布可能不同）
- 改进：基于校准的自适应阈值

**4. 与混合系统结合**：
- 当前的两步法和混合系统是独立的
- 改进：将内部表示的不确定性作为路由信号之一
- 多信号融合：SetFit MC Dropout 方差 + LLM 内部表示距离

**5. 多模型支持**：
- 当前仅在 Mistral-7B 上验证（需要开源权重）
- 改进：探索黑盒 LLM 的替代信号（如 logprobits 如果可用）

**6. 更好的离线表示存储**：
- 当前存储所有训练样本的表示
- 改进：聚类后存储类中心，减少比较开销
- 改进：使用 ANN（近似最近邻）加速大规模检索

</details>

### Q14: 本文的负面数据增强方法有哪些可能的失败场景？如何改进？

<details>
<summary>查看答案</summary>

**可能的失败场景**：

**1. 关键词提取不准**：
- KeyBERT 可能提取不到真正的关键词
- 例："I need help" → KeyBERT 可能提取不到 "help" 为关键词
- 删除非关键词可能不产生有意义的 OOS 样本

**2. 增强样本过于简单**：
- 简单删除关键词可能不够"欺骗性"
- 模型可能轻松识别这些人为生成的 OOS 样本
- 真实 OOS 查询的分布可能完全不同

**3. 短查询场景**：
- 只有几个词的查询，删除关键词后可能变成空字符串
- 替换后失去所有语义信息

**4. 与真实 OOS 分布不匹配**：
- 增强样本的分布（残缺的真实查询）与真实 OOS 分布（完全不相关的查询）差异大
- 模型可能只学会了识别"残缺查询"而非真正的 OOS

**改进方向**：

1. **更智能的增强策略**：
   - 使用 LLM 生成更逼真的 OOS 查询（知识蒸馏方式）
   - 对抗性增强：生成最"容易混淆"的 OOS 样本

2. **分层增强**：
   - 对不同长度的查询使用不同的增强策略
   - 短查询用同义词替换，长查询用关键词删除

3. **动态增强**：
   - 在训练过程中逐步增加增强样本的难度
   - 课程学习（Curriculum Learning）思路

4. **真实 OOS 样本收集**：
   - 从部署系统中收集被拒绝的查询
   - 使用无监督 OOS 检测方法从未标注数据中挖掘

</details>

### Q15: 如何将本文的不确定性路由思想推广到其他 NLP/NLU 任务？

<details>
<summary>查看答案</summary>

**可推广的任务场景**：

**1. 情感分析**：
- 轻量模型处理明确的正面/负面
- 模糊、讽刺、双关的文本路由到 LLM
- MC Dropout 在情感边界案例上的不确定性应该更高

**2. 命名实体识别（NER）**：
- 规则 + CRF 处理高频实体
- 罕见实体、新实体路由到 LLM
- 实体边界歧义可以作为不确定性信号

**3. 机器翻译**：
- 简单句子用轻量翻译模型
- 复杂句式、习语、文化特定表达路由到 LLM
- 翻译质量估计可以作为路由信号

**4. 代码审查**：
- 静态分析工具处理明确的 bug
- 复杂的设计问题、安全漏洞路由到 LLM
- 分析工具的置信度分数作为路由依据
