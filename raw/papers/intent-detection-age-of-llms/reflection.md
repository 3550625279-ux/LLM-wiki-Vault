# 反思与扩展：Intent Detection in the Age of LLMs

---

## 🔭 如果我要扩展这篇论文...

### 方向 1：更智能的路由策略

**现状**：使用 MC Dropout 方差作为唯一的路由信号。

**扩展思路**：
- **多信号融合**：组合 SetFit 置信度 + 查询长度 + 查询稀有度 + 历史路由模式
- **强化学习路由**：将路由决策建模为 MDP，用 RL 学习最优路由策略（类似 active learning 的思路）
- **自适应阈值**：根据查询类型动态调整路由阈值（例如：面向客户的查询更保守，内部测试更激进）

### 方向 2：LLM 生成合成数据训练 SetFit（知识蒸馏）

**现状**：论文在 Limitations 中提到但未探索。

**扩展思路**：
- 用 LLM 为每个意图生成多样化的训练查询
- 用 LLM 的 CoT 推理作为"软标签"训练 SetFit
- 这可能比混合路由更"干净"——不需要在线调用 LLM

### 方向 3：多语言意图检测

**现状**：仅评估英语。

**扩展思路**：
- LLM 天然支持多语言，但 SetFit 需要多语言微调
- 可以用 LLM 的多语言能力为其他语言生成训练数据
- 混合系统的路由行为在不同语言间可能不同（低资源语言可能更多路由到 LLM）

## ❓ 开放问题

### 1. LLM 的 OOS 检测能力能否从根本上提升？

论文揭示了 LLM OOS 检测差的表象，但根本原因是什么？可能的解释：
- LLM 训练时被优化为"总是给出答案"，不善于说"不知道"
- RLHF 可能强化了这种倾向（用户不喜欢"我不知道"的回答）
- 是否可以通过专门的微调（OOS-aware fine-tuning）来改善？

### 2. 路由决策的可解释性

当前路由决策是"黑盒"的（基于 MC Dropout 方差），用户无法理解为什么某些查询被路由到 LLM。是否可以：
- 提供路由决策的解释
- 让 bot 开发者理解哪些类型的查询会触发 LLM
- 基于解释优化意图设计

### 3. 动态意图体系

现实中的 TODS 意图不是静态的——新的意图会加入，旧的意图会过时。如何设计一个能自适应变化的混合系统？

### 4. 多轮对话中的意图检测

本文聚焦于单轮查询的意图检测。在多轮对话中，上下文对意图识别至关重要：
- LLM 的长上下文能力是否在多轮场景下更有优势？
- 混合系统如何处理上下文？

## 🤔 脆弱的假设

### 假设 1：SetFit 的不确定性是好的路由信号

**风险**：MC Dropout 的不确定性估计是近似的，可能在某些情况下：
- 过于保守（简单查询也被路由到 LLM）
- 过于激进（复杂查询被 SetFit 直接处理）
- 对 SetFit 模型的架构和超参数敏感

### 假设 2：负面增强样本代表 OOS 分布

**风险**：删除/替换关键词生成的 OOS 样本与真实 OOS 查询差异很大。真实 OOS 查询可能是完全不相关的话题，而非"残缺的 in-scope 查询"。

### 假设 3：内部表示的可迁移性

**风险**：论文仅在 Mistral-7B 上验证。不同 LLM 的内部表示空间可能有本质差异。对于闭源 LLM，无法获取内部表示。

### 假设 4：评估数据集的代表性

**风险**：6 个数据集全部来自购物领域。在其他领域（医疗、法律、教育）的泛化能力未验证。

## 🚧 实践中可能遇到的问题

1. **MC Dropout 与 batch inference 的冲突**：Dropout 要求逐样本推理，但生产环境通常用 batch 来提升吞吐量
2. **LLM API 成本波动**：路由到 LLM 的查询比例可能随时间和业务变化，成本不可预测
3. **ICL 示例的时效性**：训练数据中的示例可能过时（产品更新、政策变化），需要定期更新向量库
4. **标签描述的质量**：由 Claude v3 Sonnet 生成的标签描述可能不够准确或不够全面
5. **阈值调优的循环依赖**：路由阈值需要在生产数据上校准，但生产数据需要系统先部署

## 💭 个人思考

这篇论文最大的价值不在于某个单一的创新，而在于它展示了一种**工程化的思维方式**：

1. **不追求单一模型的极致**，而是用系统设计弥补单个组件的不足
2. **通过控制实验理解模型行为**，而非盲目调参
3. **关注部署约束**（延迟、成本），而非只看离线指标

这对于工业界的 NLU 系统设计非常有参考价值。核心启示：**与其花大量精力调一个"完美模型"，不如设计一个"聪明的系统"**。
