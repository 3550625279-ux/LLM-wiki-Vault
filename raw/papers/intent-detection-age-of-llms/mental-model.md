# 思维框架：Intent Detection in the Age of LLMs

---

## 🧠 这是什么类型的问题？

**意图检测**本质上是一个**文本分类**问题，但在 TODS 场景下有独特的挑战：

```
标准文本分类:
  输入文本 → 分类器 → 预测类别

TODS 意图检测:
  用户查询 → 分类器 → 预测类别（in-scope）或拒绝（OOS）
                         ↓
                   还要考虑：类别范围差异大、数据不平衡、少样本
```

**更准确地说**，这是一个**开放集识别（Open Set Recognition）**问题：系统不仅要在已知类别中分类，还要能识别不属于任何已知类别的输入。

## 🎯 如何在脑中分类这篇论文

**一句话分类**：这是一篇**工程实践导向的系统论文**，核心贡献是提出了一个**可部署的混合意图检测系统**，并附带了对 LLM OOS 检测能力的深入分析。

**三个关键词**：
1. **Hybrid** — LLM + 轻量模型的混合路由
2. **Uncertainty** — 基于 MC Dropout 的不确定性估计
3. **OOS Detection** — LLM 的域外检测弱点和改进

**类比理解**：
- SetFit 像**门诊分诊台**：快速判断，但复杂的会看走眼
- LLM 像**专科医生**：准确但排队长、费用高
- 混合系统像**智能分诊**：简单问题分诊台直接处理，疑难杂症转专科

## 📐 思维模型：不确定性路由

这是本文最核心的思想，值得内化为一个通用模式：

```
任何 "轻量模型快但不够准 + 重量模型准但太慢" 的场景都可以用这个框架：

┌─────────────────────────────┐
│    快速模型（低延迟）        │──→ 置信度高 → 直接输出
│                             │
│    置信度低 → 调用慢速模型   │──→ 输出结果
└─────────────────────────────┘

关键问题：
1. 如何定义"置信度"？（本文用 MC Dropout 方差）
2. 路由阈值如何设？（与业务需求平衡）
3. 两个模型如何互补？（不同错误模式）
```

**可迁移场景**：
- 图像分类：轻量 CNN + 大模型 Vision Transformer
- 代码审查：规则检查 + LLM 深度分析
- 搜索排序：BM25 + 语义重排序
